# Windows環境で Codex に文字化けした情報を読ませたり書かせたりしないためのベストプラクティス - 環境整備より「こう指示する」で事故を減らす

Windows で Codex に日本語を含むファイルを扱わせるとき、最初に効くのは エディタや shell の設定を全部そろえることよりも、**Codex に「どう読んで、どう書いて、どこで止まるか」を明示すること**です。

特に困りやすいのは、次のような場面です。

- UTF-8、CP932、UTF-16 系のファイルが混在している
- 見た目は読めているように見えるが、実バイトの解釈がずれている
- 既存ファイルを少し直しただけのつもりが、保存の瞬間に別エンコーディングで再保存してしまう
- CSV、TXT、ログ、Markdown、設定ファイルのような「コード以外」で壊れる
- PowerShell や一時スクリプトで雑に出力した結果、そのまま事故が固定化する

OpenAI の Codex ベストプラクティスでも、Codex は単発のアシスタントというより、設定と作業ルールを与えて継続的に使う「チームメイト」に近いものとして扱うと効果が出やすい、と整理されています[^best-practices]。また、Codex は `AGENTS.md` を作業前に読むため、毎回同じ注意点を口頭で繰り返すより、**文字コードに関するルールを常設したほうが安定しやすい**です[^agents-md]。

この記事では、**Windows で Codex に日本語ファイルを安全に扱わせるために、最初に与えると効きやすい指示**を、そのままコピペできる形で整理します。  
重点は環境構築ではなく、次の 3 点です。

- **推測で再保存しない**
- **既存ファイルのエンコーディングを勝手に変えない**
- **書いたら再読込して、日本語が壊れていないことを確認する**

## 1. まず結論

結論から言うと、Windows 環境で Codex の文字化け事故を減らすうえで一番効きやすいのは、**「文字コードの作業手順」を先に固定すること**です。

特に効くのは、だいたい次のルールです。

- 日本語を含む既存ファイルは、**読む前にエンコーディング候補、BOM 有無、改行コードを確認させる**
- 文字化けが疑われるファイルは、**自信が持てるまで保存させない**
- 既存ファイルは、**元のエンコーディング、BOM、有無、改行を維持させる**
- 新規ファイルは、**リポジトリ規約に従って UTF-8 系へ寄せる**
- 書き込みは、**エンコーディングを明示できる方法だけを使わせる**
- 書いたあとは、**再読込して日本語の代表行を検証させる**

実務での短い言い方にすると、ほぼこれです。

- **読ませる前に確認**
- **怪しいなら保存禁止**
- **既存は維持、新規だけ UTF-8**
- **曖昧な書き込みを禁止**
- **最後に再読込して確認**

逆に、事故を起こしやすい指示はこうです。

- 「文字化け直して」
- 「全部 UTF-8 にして」
- 「CSV 出して」
- 「変なところは適当に合わせて」
- 「とりあえず保存して見て」

これらはどれも、**Codex がどの段階で止まるべきか**が書かれていません。  
文字化け対策では、何をするかだけでなく、**どこで保存を止めるか**まで指示する必要があります。

## 2. なぜ Windows で文字化け事故が起きやすいのか

これは別に「Codex が日本語に弱い」という話ではありません。  
多くの場合の本当の問題は、**人間側の Windows 資産に、複数の文字コードと複数の書き込み経路が共存している**ことです。

Windows の実務では、次のような混在が珍しくありません。

- 新しめのソースや Markdown は UTF-8
- 古い CSV、TXT、ログ、設定、バッチ類は CP932 系
- 一部の出力やツール生成物は UTF-16 系
- エディタ、shell、社内ツール、Excel 由来の出力で保存経路がばらばら
- 改行コードも LF と CRLF が混在

この状態で Codex が 1 回でも間違った解釈をすると、**読めていない文字列を「読めているもの」として次の編集に進む**ことがあります。  
そして、そのまま保存してしまうと、今度は「表示上の誤解」ではなく、**ファイル自体の破損**として固定されます。

ここで大事なのは、文字化け対策は「生成品質」の話だけではなく、**I/O 手順の管理**だということです。

- どう読むか
- どの条件なら保存してよいか
- どの条件なら止まるべきか
- 何をもって「壊れていない」と見なすか

この 4 点が曖昧だと、どんなに賢いモデルでも事故は起きます。  
逆に言えば、ここをはっきり書けば、Windows でもかなり安定します。

## 3. 最初に Codex へ固定したい 6 つのルール

### 3.1 読む前に、エンコーディング候補と BOM と改行を確認させる

最初のルールはこれです。

> 日本語を含む既存ファイルを読む前に、現在のエンコーディング候補、BOM 有無、改行コードを確認し、怪しければそのまま内容解釈に進まないこと。

ポイントは、**「テキストを見る前に、まずファイルの前提を見る」**に変えることです。

Codex への指示文としては、たとえば次のように書くと効きやすいです。

```text
日本語を含む既存ファイルを読む前に、
- encoding 候補
- BOM の有無
- 改行コード
を確認してください。怪しい場合は、内容の要約や修正に進まず、
まず候補と根拠を報告してください。
```

### 3.2 文字化けが疑わしいファイルは、推測のまま保存させない

これが一番重要です。

> 文字化けが疑われるときは、調査段階では read-only とし、解釈に自信が持てるまで上書き禁止にする。

人間でも同じですが、**読めていないファイルを保存してはいけません**。  
「少し壊れて見えるけど、たぶんこれだろう」で保存すると、それが事故の確定版になります。

Codex には、こう言っておくのが安全です。

```text
文字化けが疑われるファイルは、推測のまま保存しないでください。
まず「ファイル自体が壊れている」のか「表示側だけが壊れている」のかを切り分け、
解釈に自信が持てるまで read-only で調査してください。
```

### 3.3 既存ファイルは維持し、新規ファイルだけ UTF-8 を基本にする

文字化け対策の文脈で、意外と危ないのが「全部 UTF-8 に統一して」です。

もちろん最終的に repo 全体を UTF-8 に寄せる判断はありえます。  
ただし、それは**別タスク**として設計し、差分と影響範囲を見ながらやるほうが安全です。

日常の改修タスクでは、まず次の運用が安定します。

- **既存ファイルを編集するときは、元のエンコーディングを維持する**
- **新規ファイルを追加するときは、repo 規約に従って UTF-8 系を使う**
- **既存ファイルの変換が必要なら、通常の機能修正と分ける**

Codex への指示文は、こうです。

```text
既存ファイルは元の encoding / BOM / 改行コードを維持してください。
新規ファイルは、このリポジトリの規約に従って UTF-8 系で作成してください。
既存ファイルを UTF-8 に変換したい場合は、通常修正と混ぜず、別の明示タスクとして扱ってください。
```

### 3.4 曖昧な書き込み経路をデフォルトで使わせない

Windows で事故を増やしやすいのは、「ちょっとした出力だから shell で雑に書く」です。

たとえば、

- リダイレクトでそのまま吐く
- 便利コマンドでそのまま保存する
- 一時生成物をそのまま本番ファイルへ昇格する
- エディタ任せの save を当然視する

といった流れです。

こういう経路は、**エンコーディングが明示されていない**ことが多く、事故の温床になります。  
だから Codex には、手段の選び方も固定しておくのが安全です。

```text
書き込みは、encoding を明示できる方法だけを使ってください。
PowerShell の便利な書き込みや shell のリダイレクトをデフォルトの手段にしないでください。
既存ファイルを更新する場合は、元の encoding / BOM / 改行を保てる方法を選んでください。
```

ここで重要なのは、「PowerShell を使うな」ではありません。  
**曖昧なまま書くな**、です。

### 3.5 書いたあとに必ず再読込して、日本語の代表行を検証させる

書き込みコマンドが成功しても、それだけでは十分ではありません。  
文字化け対策では、**保存のあとにもう一回読む**ところまでが 1 セットです。

確認させたいのは、たとえば次のような点です。

- 修正前に読めていた日本語の代表行が、修正後も同じように読めるか
- `�` や `?` が不自然に増えていないか
- BOM や改行コードの意図しない変化がないか
- 差分が業務変更ではなく「文字コード事故」になっていないか

Codex には次のように指示できます。

```text
書き込み後は対象ファイルを再読込し、
- 日本語の代表行
- 変更した近辺
- BOM / 改行コード
を確認してください。
コマンド成功だけで完了にせず、文字化けがないことを検証してください。
```

### 3.6 破壊兆候を「異常」として報告させる

文字化けは、壊れてから気付くと遅いです。  
なので、**壊れ始めの兆候**を Codex に拾わせるのが有効です。

よくある兆候は、たとえば次のようなものです。

| 兆候 | まず疑うこと |
|---|---|
| `�` が出る | どこかで不正なデコードが起きている |
| `?` が不自然に増える | 変換時に置換が起きている |
| 日本語だけ差分が大量に出る | 保存時の再エンコード事故 |
| ファイル全体が差分になる | 改行コード、BOM、全体再保存の事故 |
| 見た目は直ったが別ツールで崩れる | 表示側だけ直っていて、実ファイルの互換性が崩れている |

Codex には、こう伝えるとよいです。

```text
次の兆候があれば作業完了にせず、異常として報告してください。
- 置換文字の混入
- `?` の増加
- 日本語行の大量差分
- BOM / 改行コードの意図しない変更
- 変更意図のないファイル全体差分
```

## 4. プロンプトは Goal / Context / Constraints / Done when で書く

OpenAI の Codex ベストプラクティスでは、よい初期プロンプトの型として **Goal / Context / Constraints / Done when** の 4 点が紹介されています[^prompt-structure]。  
文字化け対策でも、この型がそのまま使えます。

| 項目 | 文字化け対策で何を書くか |
|---|---|
| Goal | 何を直したいのか。例: 日本語を含む既存ファイルを壊さず修正したい |
| Context | 対象ファイル、代表文字列、既知の文字コード、Windows / PowerShell 前提など |
| Constraints | 推測保存禁止、既存 encoding 維持、明示的な書き込み方法だけ使用、再読込検証 |
| Done when | 代表文字列が壊れていない、差分が意図どおり、BOM / 改行事故なし |

この型のよいところは、**「何をしてほしいか」と「何をしてはいけないか」が分かれる**ことです。  
文字化け事故は、だいたい Constraints と Done when が弱いときに起きます。

たとえば「CSV を出力して」だけだと、Goal はあります。  
でも、

- 何の文字コードで出すのか
- 既存ファイルに合わせるのか
- 壊れていないことをどう確認するのか

が空欄です。

だから、文字化け対策では **Constraints を具体的に、Done when を検証可能に書く** のが効きます。

## 5. そのまま使える短い指示文

まずは長くしすぎず、毎回のタスクに添える短い版です。

```text
この作業では文字コード事故を最優先で避けてください。

- 日本語を含む既存ファイルは、読む前に encoding 候補、BOM 有無、改行コードを確認する
- 文字化けが疑われるファイルは、推測のまま保存しない
- 既存ファイルは元の encoding / BOM / 改行コードを維持する
- 新規ファイルは repo 規約に従って UTF-8 系で作成する
- 書き込みは encoding を明示できる方法だけを使う
- 書き込み後は再読込し、日本語の代表行が壊れていないことを確認する
- `�`、`?` の増加、ファイル全体差分、BOM / 改行事故があれば異常として報告する
```

さらに、対象ファイルが決まっているなら、次の 1 行を足すと安定します。

```text
対象ファイル: path/to/file1, path/to/file2
代表文字列: "顧客名", "処理結果", "備考", "エラー詳細"
```

**代表文字列を渡す**のはかなり効きます。  
Codex に「この日本語が壊れてはいけない」という具体的な監視点を持たせられるからです。

## 6. AGENTS.md に常設したいテンプレート

同じ注意を何度も言うくらいなら、`AGENTS.md` に入れたほうがよいです。  
OpenAI のドキュメントでも、Codex は `AGENTS.md` を読み、グローバル設定と project 単位の指示を重ねて扱います[^agents-md]。

以下は、Windows で日本語ファイルを扱う repo 向けの、かなり実用寄りのテンプレートです。

```md
# Text encoding rules for this repository

## Scope
This repository may contain Japanese text and mixed legacy file encodings.
Avoid mojibake and accidental re-encoding above all else.

## Mandatory rules
- Before reading or editing an existing text file that may contain Japanese, first determine:
  - likely encoding
  - BOM presence
  - newline style
- If mojibake is suspected, do not save the file until the encoding interpretation is credible.
- Preserve the original encoding, BOM, and newline style for existing files.
- Treat "convert to UTF-8" as a separate, explicit task. Do not mix it into normal feature or bug-fix work.
- New files should follow the repository convention. If there is no clear rule, prefer UTF-8 and state whether BOM is used.
- Do not use ambiguous write paths by default, such as shell redirection or convenience commands without explicit encoding control.
- After writing, reopen the file and verify representative Japanese lines.
- If any of the following appears, stop and report:
  - replacement characters
  - unexpected `?`
  - large diffs on Japanese lines
  - unintended BOM change
  - unintended newline conversion
  - whole-file diffs without a business reason

## Reporting format
For each changed text file, report:
- path
- detected or preserved encoding
- BOM presence
- newline style
- how verification was performed
- whether representative Japanese text remained intact
```

このテンプレートのよいところは、**「どう編集するか」ではなく「どう壊さないか」まで固定できる**ことです。

特に効くのは、次の 2 行です。

- `If mojibake is suspected, do not save the file ...`
- `Treat "convert to UTF-8" as a separate, explicit task.`

この 2 行があるだけで、雑な事故がかなり減ります。

## 7. NG 指示と OK 指示

文字化け対策では、指示の粒度が結果をかなり左右します。  
よくない指示と、改善した指示を並べるとこうです。

| NG 指示 | OK 指示 |
|---|---|
| 文字化けを直して | まず、ファイル自体の破損か表示側だけの問題かを切り分け、推測のまま保存しないでください |
| 全部 UTF-8 にして | 既存ファイルは元の encoding を維持し、新規だけ repo 規約に従って UTF-8 系にしてください。既存変換は別タスクで扱ってください |
| CSV を出力して | CSV は既存運用の encoding に合わせ、書き込み時に encoding を明示し、出力後に日本語列を再読込して確認してください |
| 読める範囲で直して | 自信が持てない箇所は保存せず、候補と根拠を報告してください |
| 適当に合わせて | BOM、改行、encoding を勝手に変えず、差分が業務変更だけになるようにしてください |
| とりあえず保存して見て | 保存前に encoding 解釈を確認し、保存後は再読込して代表文字列を検証してください |

ポイントは、**「手を動かす前の確認」と「保存後の検証」を必ず書く**ことです。

## 8. 作業別のコピペ用テンプレート

### 8.1 既存ソースや設定ファイルを修正させるとき

```text
Goal:
Windows 環境の既存ファイルを壊さず修正したい。

Context:
日本語を含むファイルがある。
対象ファイル: <paths>
代表文字列: <examples>

Constraints:
- 既存ファイルは読む前に encoding 候補、BOM 有無、改行コードを確認する
- 文字化けが疑われる場合は推測のまま保存しない
- 既存ファイルの encoding / BOM / 改行コードは維持する
- 既存ファイルの UTF-8 変換は別タスクとする
- 書き込みは encoding を明示できる方法だけを使う
- 保存後に再読込し、日本語の代表行が壊れていないことを確認する

Done when:
- 業務上の修正だけが差分に含まれている
- 代表文字列が壊れていない
- BOM / 改行事故がない
```

### 8.2 文字化け調査だけをさせるとき

```text
このタスクでは、まだ修正しないでください。
まず調査だけをしてください。

- ファイル自体が壊れているのか
- 表示側だけが壊れているのか
- どの encoding 候補があるのか
- BOM / 改行コードはどうなっているのか
- どの段階で壊れた可能性が高いのか

解釈に自信が持てるまで保存禁止です。
必要なら、候補と根拠を並べてください。
```

このテンプレートは、「直す」より先に「切り分ける」ので、事故が減りやすいです。

### 8.3 CSV / TXT / Markdown / ログを新規出力させるとき

```text
Windows で他ツールも読む前提のテキスト出力です。
出力ファイルの文字コード事故を避けてください。

- 既存出力がある場合はその encoding / BOM / 改行コードに合わせる
- 既存ルールが不明な場合は UTF-8 系を第一候補とし、採用した方針を明示する
- 書き込みは encoding を明示できる方法で行う
- 出力後に再読込して、日本語列や代表行が読めることを確認する
- `�` や `?` の混入があれば異常として報告する
```

### 8.4 「とにかく壊さない」を最優先したいときの強め版

```text
この作業では、機能修正よりも文字コード事故の回避を優先してください。

禁止事項:
- 文字化けが疑われるファイルの推測保存
- 既存ファイルの無断 UTF-8 化
- BOM / 改行コードの無断変更
- encoding を明示しない書き込み
- 代表文字列を確認しないままの完了報告

必須事項:
- 読む前の前提確認
- 保存前の解釈確認
- 保存後の再読込確認
- 異常兆候の報告
```

## 9. レビュー時のチェックリスト

Codex に作業させたあと、人間側で見るチェックポイントも固定しておくとさらに安定します。

### 9.1 Codex に報告させたい項目

- 変更したファイルの一覧
- 各ファイルの encoding / BOM / 改行コードの扱い
- 日本語の代表行をどう検証したか
- 保存前に怪しい点がなかったか
- 変換を伴う変更が入っていないか

### 9.2 差分で見るべきこと

- 日本語行だけ不自然に大きく変わっていないか
- 改行だけの差分が大量に出ていないか
- BOM の有無が変わっていないか
- 業務変更と無関係な全体差分がないか
- CSV やログで列崩れ、引用符崩れが起きていないか

### 9.3 その場で戻したくなる差分

次の差分が出たら、少なくとも一度止めたほうがよいです。

- ほぼ全文が差分
- 日本語だけ `?` だらけ
- 表示上は読めるが、別の既存ツールで崩れる
- 「変更は数行のはず」なのに、なぜか何百行も変わっている

文字化け対策で大事なのは、**成功した差分を増やすことより、怪しい差分を早く止めること**です。

## 10. それでも不安定ならどうするか

ここまでの指示を入れてもまだ不安定なら、次の順で強化するのがおすすめです。

1. **単発プロンプトを整える**
2. **`AGENTS.md` に常設する**
3. **同じ作業を `Skill` 化する**
4. **それでも残る問題だけ、editor / shell / WSL など環境側で吸収する**

OpenAI の Codex ドキュメントでも、繰り返し使うワークフローは `AGENTS.md` や Skill に寄せていく整理がされています[^best-practices]。  
また、Windows ではネイティブ利用に加えて WSL の選択肢も用意されています[^windows-doc]。

ただ、運用の優先順位としては、まず環境移行ではなく、

- **どう読ませるか**
- **どう保存させるか**
- **どこで止めるか**
- **何をもって正常とするか**

を Codex に覚えさせるほうが先です。  
文字化け事故の多くは、この手順が曖昧なことから起きるからです。

## 11. まとめ

Windows 環境で Codex に日本語ファイルを扱わせるとき、最初に効くのは「ユーザーの PC を完璧に整備すること」ではありません。  
それより先に、**Codex に文字コードの作業手順を明示すること**です。

特に覚えておきたいのは、次の 5 点です。

- **読む前に、encoding / BOM / 改行を確認させる**
- **文字化けが疑わしければ、推測のまま保存させない**
- **既存ファイルは維持し、新規ファイルだけ UTF-8 系へ寄せる**
- **曖昧な書き込み経路を禁止する**
- **保存後に再読込して、日本語の代表行を確認させる**

そして、毎回言うくらいなら `AGENTS.md` に入れる。  
これが一番実務的です。

文字化け対策は、気合いで「日本語をちゃんと扱って」と頼む話ではありません。  
**保存してよい条件と、止まるべき条件を明文化する話**です。

そこまで書けば、Windows でも Codex はかなり扱いやすくなります。
